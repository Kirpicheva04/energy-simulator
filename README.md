# energy-simulator

## Описание
Симулятор распределения энергии на 24 часа. Два тестовых сценария: летний день с избытком энергии и зимний день с дефицитом.

## Требования
- Python 3.14

## Установка и запуск
```bash
pip install -r requirements.txt
python main.py
```

## Структура проекта
- `models.py` — классы Consumer и Generator
- `optimizer.py` — логика оптимизации
- `visualizer.py` — визуализация результатов
- `test_cases.py` — тестовые сценарии (лето/зима)
- `main.py` — запуск симуляции

## Результаты
Для каждого сценария выводятся:
- Таблица работы генераторов по часам
- Таблица почасовой стоимости
- Таблица отключённых потребителей
- Графики в одном окне

## Использование ИИ
ИИ использовался для генерации кода как инструмент ускорения рутинных задач.

### 1. Генерация тестовых данных
**Промпт:** "Я пишу на Python симулятор энергосети. Нужны тестовые данные. Создай 10 потребителей: жилые дома, офис и так далее. Для каждого — профиль потребления на 24 часа (24 числа). Ночью потребление ниже, утром и вечером — пики. Офис не потребляет ночью. Создай 3 солнечные панели: зимой световой день короче и генерация ниже, чем летом."

**Что сделал ИИ:** сгенерировал профили потребления для домов, офиса, школы, больницы.

**Как проверила:**
- ночью потребление снижается
- утренний пик и вечерний пик присутствуют
- школа, офис не потребляют энергию ночью 
- больница имеет стабильно высокое потребление
- профили солнечных панелей составлены на основе логики: зимой световой день короче и генерация ниже

Также создана проверка в коде: при создании потребителя или генератора через `__post_init__` автоматически проверяется корректность переданных массивов — длина 24 элемента, отсутствие отрицательных значений и соответствие типа генератора. Некорректные данные вызывают исключение с понятным описанием ошибки.

### 2. Код визуализации
**Промпт:** "Напиши визуализацию на matplotlib: все графики в одном окне. Верхний левый — stacked bar генерация энергии по источникам с линией спроса красным пунктиром. Верхний правый — столбцы стоимости по часам с линией средней стоимости. Нижний левый — красные столбцы дефицита по часам. Нижний правый — столбцы количества отключённых потребителей по часам. Если дефицита нет — только два верхних графика в один ряд."

**Как проверила:** После запуска проверяла, что сумма столбцов stacked bar равна значениям в текстовой таблице. Настроила отступы между графиками, подписи осей и легенду для читаемости. Заменила `fill_between` на столбчатую диаграмму для не искаженного отображения почасовых данных.

### 3. Оптимизатор
**Что сделал ИИ:** написал базовую структуру оптимизатора.

**Как проверила:**
- Нашла и исправила ошибку: дизель был 15 кВт вместо 35 кВт, из-за чего даже летний сценарий показывал дефицит
- Нашла и исправила сортировку потребителей: убрала минус в `key=lambda`, чтобы мелкие потребители подключались первыми и максимизировалось количество подключённых
- Проверила логику вручную на нескольких часах: пересчитала ожидаемую генерацию и сравнила с выводом программы

### 4. Модели данных
Классы `Consumer` и `Generator` с валидацией через `__post_init__` написаны самостоятельно.

## Допущения и упрощения модели

### Текущие упрощения:
1. Нет накопителей энергии (аккумуляторов). Избыток энергии, выработанный днём солнечными панелями, не может быть сохранён и использован ночью.
2. Нет потерь при передаче. Вся сгенерированная энергия доходит до потребителей без потерь.
3. Мгновенное изменение мощности.
4. Нет случайных факторов: облачность, аварии, поломки генераторов и т.д.
5. Все потребители равны по приоритету. Хотя поле `priority` есть в модели, во всех сценариях оно равно 1. В реальности больница важнее спортзала. Это можно использовать, чтобы расширить модель.

### Возможные расширения:
1. Аккумуляторы. Добавить класс `Battery` с параметрами: ёмкость, максимальная мощность заряда, КПД. Избыток дневной солнечной энергии запасается и используется ночью.
2. Динамические ограничения. Использовать поле `ramp_rate` для учёта максимальной скорости изменения мощности генераторов.
3. Добавить случайные факторы: облачность для солнечных панелей, вероятность поломки дизеля, случайные отклонения потребления.
4. Приоритеты потребителей. Задать разные значения `priority`: больница = 1 (не отключать никогда), школа = 2, жилые дома = 3, спортзал = 4.
